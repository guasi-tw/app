# About 頁設計（關於 guasi）— Design Spec

**Date:** 2026-06-16
**Status:** Approved (content + structure locked via visual brainstorming)
**Branch / worktree:** `about-page` @ `~/Source/github/guasi-tw/about` (off `main`, isolated from `slice2-add-account-binding`)
**Route:** `/about` → `app/about/page.tsx`

---

## 1. 目的

一個公開的「關於」頁，向潛在使用者**介紹 guasi 是什麼、為什麼需要**，並在結尾**鼓勵註冊、建立自己的正身頁**。可直接分享給人看。

- **對象：** 潛在使用者（KOL／一般創作者），可能第一次接觸這個服務。
- **語氣：** 對使用者講「你的好處」，不是對內講「我們的好處」。成長／行銷是我們的副產品，**不寫進文案**。
- **裝置：** 手機優先、精簡。單欄、由上而下捲動。
- **語言：** 全繁體中文（台灣用語）。例外：品牌字 `guasi`、羅馬拼音 `guá-sī` / `tsiànn-sin`、平台名 Threads／Instagram／miin.cc。

## 2. 命名／敘事策略（重要決策）

- **主角是品牌「guasi（我是）」，不是「正身」。** 讀者要記住的是 guasi。
- 「正身」雖是產品 UI 用語（建立正身／驗明正身），但在台灣不算常用詞 → 在 About 頁**只當一行註解輕輕帶過**，附羅馬拼音 `(tsiànn-sin)`，**不做文化／建築考據、不解釋「本尊」**。
- 開場用人人都懂的一句「這真的是我」勾住讀者；`guasi = 我是`（guá-sī）就是那句話，重複以加深記憶。
- 收尾的公開查詢頁仍叫「驗明正身」，輕輕呼應「正身」第二層意思即可。

> 內容來源：`app/docs/product-pitch.md`（已有完整繁中敘事，可作文案母本）。

## 3. 頁面結構（七段，由上而下）

| 0 | 頂列 | 左 `guasi` 字標、右「關於」 |

## 4. 最終文案（繁中，鎖定）

**① 痛點鉤子**
- 標題：`帳號被封了，你要怎麼說「這真的是我」？`
- 內文：`2026 年 6 月，Meta 一波大封號，許多人一夕失去經營多年的主帳號。新帳號沒有歷史，舊帳號已無法發聲；冒名者卻搶著喊「本尊回來了」。最該證明自己的那一刻，你反而最沒辦法證明。`

**② 品牌答案**
- 小標：`一個字的答案`
- 大字：`guasi`　副：`guá-sī ·「我是」`
- 內文：`把那句你最想喊、卻沒人信的「我是本人」，變成任何人都能查證的事實。這就是 guasi（我是）。`

**③ 怎麼運作**
- 標題：`怎麼運作`
- 內文：`趁帳號還活著，在 guasi 把你散落各平台的帳號綁定、驗證、公開連結。被封時，存活的帳號就能替你的新帳號背書。`
- 註解（muted）：`（你的身分中樞，我們叫它「正身」(tsiànn-sin)；各平台帳號則是「分身」。）`
- 三步驟：
  1. `建立正身`　`註冊一個帳號，當你所有分身的中樞。`
  2. `註冊分身`　`複製一段含驗證碼的文字，從該帳號發一篇公開貼文，貼回網址即完成。`
  3. `驗明正身`　`任何人免登入就能查：這個帳號，跟哪些帳號是同一個人。`
- 範例貼文下方註解：`↑ 這篇公開貼文本身，就替你向所有看到的人驗明正身。`

**支援平台列**
- 小標：`目前支援`
- chips：`Threads`、`Instagram`、`miin.cc`、`更多陸續支援`（虛線樣式）

**④ 驗明正身的公開頁**
- 標題：`驗明正身的公開頁`
- 說明：`任何人輸入帳號，就能看到這個人公開承認的其他帳號與驗證證據。`
- 卡片：標示「範例」，含頭像、`小美`、`guasi.tw/gua/meimei`，三筆帳號（Threads @meimei、Instagram @meimei.ig、Threads·新帳號 @meimei.new），各帶 `✔ 已驗證 · YYYY/MM`。

**⑤ 為什麼可信**
- `✔ 驗證靠本人公開動作`——`貼文必須從該帳號本人發出，冒名者沒有控制權就發不出來。`
- `✔ 證據是公開貼文，人人可查`——`每筆綁定都附上原始驗證貼文連結，任何人都能點開核對，不必只相信我們。`
- `✔ 不靠平台授權`——`畢竟會封人的正是平台；身分的可信度握在你自己手上。`

**⑥ 為什麼是現在**
- 標題：`為什麼是現在`
- 內文：`封號當下，被封的帳號已經發不出驗證貼文。guasi 的價值來自「事先」——趁帳號還活著就登記、串好，被封後才有存活的帳號替你的新帳號背書。`

**⑦ CTA**
- 標題：`趁現在，先驗明你的正身`
- 副：`被封後就來不及了。建立正身只要幾分鐘。`
- 按鈕：`以 Google 登入，建立我的正身 →` → `/login`
- 按鈕下：`免費 · 無需密碼`
- footer：`guasi.tw　·　我是正身`

範例貼文文字（照真實截圖）：`@gua.si.tw` / `我是分身認證貼文` / `點此觀看此帳號的正身：` / `guasi.tw/r/ABC…` / `我是分身驗證碼：ABCDEF`；連結預覽卡 `🌐 guasi.tw` / `我是正身 · guasi`。

## 5. 與目前實作的一致性（鎖定的正確性約束）

- **登入只有 Google**（`app/(auth)/login/page.tsx` 只 `signIn("google")`）→ CTA 不提 Email magic-link（雖為 locked 計畫，但尚未實作）。
- **沒有截圖快照**（`prisma/schema.prisma` ProofRecord 只存 `proofPostUrl`；snapshot 欄位 MVP 未用，§A.1）→ 可信度文案**只講「公開貼文連結，人人可查」，不講「存截圖／封了還能查」**。封號後的存活價值來自**跨帳號背書**，不是死掉的貼文。
- **正身公開頁是 Slice 3**（`app/gua/[slug]` 目前為 stub）→ About 頁的公開頁卡是**靜態「範例」插圖**，明確標示。待 Slice 3 完成後可換成真實截圖或連結。範例不要承諾尚未定案的 UI 細節。
- CTA 流程：`/about` → 點按鈕 → `/login`（Google）→ `/onboarding`，與現況一致。

## 6. 技術實作

- **新增單一路由：** `app/about/page.tsx`（Server Component，純靜態、無資料抓取）。**純加法**，不動既有檔案邏輯。
- **樣式：** 沿用 `app/globals.css` 既有 token（`--bg #0b0b0f`、`--fg`、`--muted`、`--accent #e8b500`、`--fg` 字體堆疊）。新增 about 專屬 class（前綴 `about-`）集中放在 `globals.css` 末段（如 `/* --- About page --- */` 區塊），避免與 slice2 衝突。
- **版面：** 單欄置中，內容寬上限約 `36rem`，手機優先；段落以細分隔線（`#16161d`）分隔；品牌段與 CTA 段用深金漸層底（`linear-gradient(180deg,#15130a,#0b0b0f)`）。
- **無新依賴**、無 client JS（純標記 + CSS）。頭像用 CSS 漸層占位即可（範例插圖）。
- **Metadata：** 為 `/about` 設 `export const metadata`（title/description），利於分享。
- **連結：** 首頁可加一個指向 `/about` 的入口（可選，列為次要 todo，不阻擋本頁上線）。

## 7. 範圍外（YAGNI）

- 不做 i18n／多語切換。
- 不做動畫／互動（純靜態頁）。
- 不接真實 `gua/[slug]` 資料（Slice 3 再說）。
- 不寫 Email 登入相關文案。
- 不做截圖快照相關承諾。

## 8. 驗收

- `/about` 在手機寬度（375px）下單欄、無水平捲動、字級可讀。
- 全繁中（除品牌字／拼音／平台名）。
- CTA 連到 `/login` 並可完成 Google 登入流程。
- 範例貼文與範例公開頁都標示「範例」。
- 文案與第 5 節的正確性約束一致。
- `npm test`、`npm run build` 通過。
